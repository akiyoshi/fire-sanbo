# FIRE参謀 v4.0 — 結果画面の情報階層化 + 取り崩し最適化UI接続

> **ステータス**: 承認済み（CEO縮小 + Eng v2 + Designレビュー適用）
> **作成日**: 2026-04-15
> **前提ドキュメント**: DESIGN.md (v3.0.0)

## 概要

FIRE参謀を「シミュレーター」から「アクションプランナー」へ再定義する。
画面数を増やさず、結果画面の情報階層（1軍/2軍）を導入し、
死蔵していた取り崩し最適化エンジンをUIに接続する。

## 問題

作者自身がFIRE計画を検討するツールとして使っているが、以下の痛みがある：

1. **結果画面の情報過多** — 成功率・チャート・処方箋・最悪ケース・What-if・税金内訳・取り崩し順序が1画面に同格で同居。重要な情報が埋もれ、細かいオプションを見なくなった
2. **テンプレートのフィードバック不足** — 適用しても何が変わったかわからない
3. **取り崩し最適化の死蔵** — エンジン（24通り全評価）は完成しているがUIに未接続。結果画面にハードコードされた順序が表示されるだけ
4. **計算結果がアクションに繋がらない** — アセットアロケーション見直し、積立先変更、取り崩し方法の最適化など、具体的な次の一手が示されない

## ターゲットユーザー

日本在住、FIREを真剣に検討中のITエンジニア。税制・年金の仕組みは理解しており、シミュレーションの精度を求めるが、操作に迷いたくない。月1回程度パラメータを見直して、今の計画の健全性を確認したい。

## 設計判断

### なぜ画面を増やさないか（CEOレビュー縮小モード）

- **3画面分割を撤回**: ダッシュボード→アクションプランのタブ切替は、1人ユーザーにとって摩擦が増えるだけ
- **AppState変更不要**: state machineを触らない方がリスクが低い。5フェーズ構成をそのまま維持
- **新規画面ゼロ**: results.tsx内のセクション順序と表示方法（折りたたみ）の変更で解決

### なぜ中央値シナリオか（Engレビュー v2）

- **フルモンテカルロを撤回**: 24通り×1000試行=24,000回はWorker競合を生む
- **中央値シナリオ（deterministic）**: 24通り×1試行(stdDev=0)=24回 → <50ms、Worker不要
- **同期計算**: useMemoで即時計算。キャッシュ・遅延実行・generationRefガード一切不要

### 2軍の視覚設計（デザインレビュー）

- **2軍はフラットレイアウト**: `<details>`内にCardを入れず、二重ボーダーを回避
- **セクション見出し**: `h2`「アクション」+ `hr`で1軍と2軍を視覚的に区切る
- **既存パターン踏襲**: wizard.tsxの`<details>` + ChevronRight + `group-open:rotate-90`を結果画面にも使用

## 技術的考慮事項

### 取り崩し最適化: 中央値シナリオ（決定論的）

```typescript
// optimizer.ts に OptimizeOptions 追加
export interface OptimizeOptions {
  deterministic?: boolean;  // true: numTrials=1, stdDev=0
}

// results.tsx での呼び出し — Worker不要、useMemoで同期計算
const withdrawalResult = useMemo(() => {
  const input = formToSimulationInput(form);
  return optimizeWithdrawalOrder(input, { deterministic: true });
}, [form]);
```

- **24通り × 1試行 = <50ms**: メインスレッドで即時計算
- **Worker変更不要**: simulation/worker.ts, use-simulation.ts は一切変更しない
- **cashは末尾固定**: 順列から除外し、4!=24通りを上限に保証
- **比較指標**: 最終資産（決定論的では成功率が全パターン同一のため）
